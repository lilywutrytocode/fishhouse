# 低 token 维护工作流

v0.1-mvp 之后,本项目进入**维护优先、低 token**模式。后续每一轮协作都必须遵守下列纪律,
目的是减少无谓的上下文消耗,保持改动小而可控。

## 硬性纪律

1. **每轮只做一个小任务**。不要把多个功能塞进一轮;一个任务 = 一个可独立 commit 的改动。
2. **先给 patch plan 再动手**。开工前先用几行列出:要改哪些文件、改什么、预期影响,
   等确认后再写代码。不要一上来就大改。
3. **不全项目扫描**。不要 `grep`/`read` 整个仓库。只打开与当前任务直接相关的文件;
   已知路径直接定位,不做发散式搜索。
4. **不长报告**。结果汇报默认 ≤20 行,不 dump 全量 JSON,不贴长 diff,不写长篇双语。
5. **默认只跑相关测试**。改哪个模块跑哪个模块的测试;只有在明确需要、且用户同意时,
   最后才跑一次全量测试。
6. **不自动进入下一步**。完成当前任务即停,等用户指示下一步;不要主动开新任务。

## 推荐节奏(每轮)

```
1) 复述任务(1-2 行)
2) patch plan(改哪些文件 / 改什么 / 影响)   ← 等确认
3) 最小改动
4) 只跑相关测试 / 必要时 smoke test
5) ≤20 行结果汇报,然后停
```

## 不要做的事

- 不主动做 #5 真三级共振样本、不做回测统计、不做 dashboard —— 除非用户明确要。
- 不改 `chanlun` 核心算法,除非任务本身就是改核心(且先给 plan)。
- 不为「顺手」而扩大改动范围。
- 不每一步都跑全量测试。
